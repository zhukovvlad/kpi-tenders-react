# Дневник разработки — react-kpi-tenders

## 2026-04-30 — День 8: Адаптация Extraction Wizard под extraction_requests API

### Контекст

Бэкенд переехал с «одна задача на документ» на новую сущность `extraction_requests` (см. `go-kpi-tenders/docs/devlog/2026-04-30.md`). Прежний контракт `POST /:id/extract → { task_id }` + поллинг `tasksApi.getByDocuments` больше не отражает реальную модель: один документ может обрабатываться несколькими параллельными запросами, ответы у каждого свои, а стадии `convert`/`anonymize` стали shared-артефактами и переиспользуются между запросами.

Дополнительно — пользователь должен иметь возможность отключить анонимизацию для доверенных документов (по умолчанию включена).

Цель сессии — переписать Extraction Wizard под новый API, не трогая остальную часть приложения.

---

### Новый контракт API

```http
POST /api/v1/documents/:id/extract
  body: { questions: string[], anonymize?: boolean }   // anonymize default = true
  →    { extraction_request_id: string, status: ExtractionRequestStatus }

GET  /api/v1/extraction-requests/:id
  →    ExtractionRequest { status, anonymize, questions, resolved_schema?, answers?, error_message?, ... }
```

Статусы запроса: `pending → running → completed | failed`. `resolved_schema` появляется после стадии `resolve_keys`, `answers` — после `extract`.

### API-сервис

**`src/services/api/extraction.ts`** — два метода вместо одного:

```ts
export const extractionApi = {
  initiate: (documentId: string, payload: InitiateExtractionPayload) =>
    apiClient.post<InitiateExtractionResponse>(
      `/api/v1/documents/${documentId}/extract`,
      payload,
    ).then((r) => r.data),

  get: (extractionRequestId: string) =>
    apiClient.get<ExtractionRequest>(
      `/api/v1/extraction-requests/${extractionRequestId}`,
    ).then((r) => r.data),
}
```

`initiate` теперь принимает не массив строк, а payload-объект (расширяется без breaking change при добавлении новых опций).

### `StepReviewAndRun` — поллинг через TanStack Query

Старый код:
- `useState<Task[]>` + `setInterval` 3 c → `tasksApi.getByDocument`
- Результат складывался из `extractTask.result_payload`
- 4 фиксированные стадии, статус каждой считался по `task.status`

Новый код:
- `useMutation` запускает `extractionApi.initiate(documentId, { questions, anonymize })`
- Возвращённый `extraction_request_id` сохраняется в локальном state, и активируется `useQuery`:

```ts
const { data: request } = useQuery({
  queryKey: ["extraction-request", requestId],
  queryFn: () => extractionApi.get(requestId!),
  enabled: !!requestId,
  refetchInterval: (query) => {
    const status = query.state.data?.status
    return status === "completed" || status === "failed" ? false : 3000
  },
})
```

Поллинг останавливается автоматически на терминальном статусе — никаких `useRef<ReturnType<typeof setInterval>>` и `clearInterval` в `useEffect` cleanup'е. Больше нет утечек таймеров и race-condition'ов при unmount.

### UX-маппинг стадий 3-в-1

Бэкенд внутри гонит `convert → [anonymize] → resolve_keys → extract`, но наружу выдаёт один `status`. Пытаться показать 4 шага через polling status'а нечестно — стадии `convert`/`anonymize` обе скрываются за `pending`. Поэтому UI свёрнут в 3 стадии:

```ts
const STAGES = [
  { key: "prep",    label: "Подготовка",       description: "Конвертация и защита данных" },
  { key: "schema",  label: "Анализ структуры", description: "Маппинг параметров" },
  { key: "extract", label: "Извлечение",       description: "Сбор значений" },
]

// Маппинг:
//   pending                          → stage 0 active
//   running, !resolved_schema        → stage 1 active
//   running, resolved_schema present → stage 2 active
//   completed                        → all done
//   failed                           → текущая стадия как failed
```

Деривация выглядит так:

```ts
function deriveStages(request: ExtractionRequest | undefined) {
  if (!request) return { current: -1, statuses: ["idle","idle","idle"], percent: 0 }
  const failed = request.status === "failed"
  const hasSchema = (request.resolved_schema?.length ?? 0) > 0
  const completed = request.status === "completed"

  let current: number
  if (completed) current = 3
  else if (request.status === "running" && hasSchema) current = 2
  else if (request.status === "running") current = 1
  else current = 0

  // ... + расчёт statuses[] и percent
}
```

Это не теряет полезной информации (всё равно у нас не было точечного прогресса между convert и anonymize в исходном коде — стадия определялась по наличию `task` записи), но делает UX честнее.

### Ловушки и особенности

#### `Intl.PluralRules` и строгий маппинг

Тип `Intl.LDMLPluralRule` включает все 6 значений (`zero | one | two | few | many | other`), но русский использует только 3. Прямой объектный литерал `{ one, few, many, other }[r]` ломал сборку:

```text
TS2339: Property 'two' does not exist on type '{ one: ...; few: ...; many: ...; other: ... }'
TS2339: Property 'zero' does not exist on type '...'
```

Исправление — `Partial<Record<...>>` с явным fallback:

```ts
const map: Partial<Record<Intl.LDMLPluralRule, string>> = {
  one: "параметр",
  few: "параметра",
  many: "параметров",
  other: "параметров",
}
return map[r] ?? "параметров"
```

#### Никакого больше `tasksApi.getByDocument` в визарде

Старая страница тянула задачи через batch-эндпойнт `tasksApi.getByDocuments` (тот, что мы выкатили в День 7 для `/anonymization`). Теперь у визарда отдельный канал `extractionApi.get(id)` — таски модулей `convert`/`anonymize` шарятся между запросами, и подсветка стадий не должна на них завязываться. `tasksApi` остаётся для `/anonymization`.

#### Polling stop на terminal

В TanStack Query `refetchInterval` может быть функцией, которая получает `query` и возвращает либо число (мс), либо `false` (стоп). Это идиоматичная замена ручному `clearInterval` на основе предыдущего state. Закладывая логику в саму функцию, мы избегаем лишних `useEffect`'ов и побочных эффектов.
