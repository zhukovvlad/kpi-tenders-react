# Дневник разработки — react-kpi-tenders

## 2026-05-01 — День 9: Полная переделка UX/UI и переход на IA Tender Analysis

### Контекст

До этой сессии фронтенд жил в наследии MVP: «дашборд-плитки модулей», отдельные роуты `/documents`, `/anonymization`, `/extraction`, многошаговый Extraction Wizard и тёмное glassmorphism-оформление. В дизайн-материалах (`Design/Tender_Analysis_design_tokens.md`, `Tender_Analysis_IA_routes.md`, HTML-моки) описана другая модель: продукт строится вокруг иерархии «объект → договор → параметры», язык интерфейса плотный и спокойный, а ключевые сценарии — извлечение параметров, сравнение договоров, мониторинг удорожания.

Цель сессии — переписать большую часть фронтенда под новую IA, внедрить дизайн-систему через Tailwind v4, уйти от визарда и подготовить площадку для подключения новых эндпоинтов через mock-слой.

Бекенд (`go-kpi-tenders`) пока не поднимает большинство нужных эндпоинтов, поэтому работали в режиме «UI-first»: делегируем сервисы в моки за флагом `VITE_USE_MOCKS`.

### Дизайн-токены и тема

`src/index.css` переписан под Tailwind v4. Структура:

- Базовые CSS-переменные (`--bg-page`, `--text-primary`, `--accent-primary`, `--action-primary`, `--warning`, `--neutral-*`, `--danger`, `--info`, радиусы, шрифты, тени) определены в `:root, [data-theme="light"]` и `[data-theme="dark"]`.
- В `@theme inline` они проброшены в Tailwind-утилиты как семантические цвета: `bg-page`, `bg-surface`, `bg-surface-sunken`, `bg-section-header`, `text-fg`, `text-fg-secondary`, `text-fg-tertiary`, `text-fg-muted`, `border-border-subtle/default/strong`, `bg-accent-soft`, `text-accent-text`, `bg-action`, `text-action-text`, `bg-warning-soft`, `bg-neutral-soft` и т.д.
- Параллельно поддерживаются shadcn-алиасы (`--color-background`, `--color-primary` и т.д.), смаппленные на наши токены — без них пришлось бы переписывать примитивы из `src/components/ui/`.
- `next-themes` сконфигурирован с `attribute="data-theme"` (а не `class`), чтобы дерево CSS-переменных переключалось через атрибут на `<html>`.
- В `index.html` дефолтная тёмная тема (`<html lang="ru" data-theme="dark">`).
- Hairline-бордер `0.5px` сделан дефолтом для всех Tailwind-utility `border-*` через `@layer base` — глобально перебивает `1px`.
- Добавлены кастомные утилиты через Tailwind v4 `@utility`: `container-page`, `hairline`, `hairline-strong`, `hairline-dashed`, `focus-ring`.
- Новый шрифтовой стек: `Inter / Manrope / Geist Variable` для sans, `Cormorant Garamond` для serif (использовать только на главных H1 продуктовых экранов и на лендинге).

Ключевое решение по конфликту имён: в дизайн-документе `text-primary` означает основной цвет тела, в shadcn — primary-CTA. Чтобы не ломать shadcn-примитивы, в проекте основной текст называется `text-fg` (от `--color-fg`), а `text-primary` оставлен за CTA-стилем shadcn.

### Поведение, заслуживающее упоминания

- **Темы.** Иерархия глубины строится на бордерах и подложках, не на тенях. Тени допустимы только для `popover/modal/focus`. На светлой теме `accent-soft` поднимает элемент над поверхностью; на тёмной — углубляет (это сознательная асимметрия восприятия).
- **Веса шрифта.** Только `regular (400)` и `medium (500)`. `font-bold (700)` запрещён в `index.css` — заголовки в `@layer base` принудительно `font-weight: 500`.
- **Пилюли статусов.** Единая схема `dot + text` с soft-фоном и тонким бордером — `StatusPill`. Цвет «выше порога удорожания» совпадает с цветом «max в сравнении» намеренно.
- **Polling.** На `ContractPage`, `ExtractionRequestPage` и `ExtractionRequestsList` используется TanStack Query `refetchInterval: (q) => isActive ? 1500 : false`. Polling останавливается на терминальном статусе автоматически.
- **Хлебные крошки.** На карточке объекта — короткое `<Link>… к списку объектов`. Переход назад через `useNavigate` не используется, чтобы deep-link корректно работал из истории.

## Поздний апдейт: вложенность объектов в роуте `/sites/:siteId`

В новой редакции `Tender_Analysis_IA_routes.md` (ver1) уточнено поведение роутинга вокруг иерархии. Реализовано в этой же сессии после первой части дня.

### Что было

- `/dashboard` показывал плоский список **всех** объектов, включая дочерние очереди и корпуса (с хлебными крошками от корня в карточке).
- `/sites/:siteId` всегда рендерил карточку с табами, независимо от наличия дочерних объектов.
- `/sites/:siteId/edit` отсутствовал — кнопка «Редактировать» была disabled.
- Хлебных крошек на карточке объекта не было; вместо них — статичная ссылка «← К списку объектов».
- В `/sites/new` нельзя было предзаполнить родителя.

### Что стало

- **Дашборд = только корни.** `sitesApi.listForDashboard()` фильтрует моки по `parent_id === null`. На стороне бэкенда отражено как `?aggregate=true&roots=true`. Дочерние очереди и корпуса доступны через drill-down с промежуточного экрана родителя.
- **`/sites/:siteId` = два поведения по одному роуту.** В `SitePage` подгружается общий список объектов (`sitesApi.list`), вычисляется `hasChildren` и на его основе рендерится либо новый `SiteOverview` (промежуточный экран), либо прежний `SiteCardView` с табами.
- **Промежуточный экран `SiteOverview`** (`src/components/site/SiteOverview.tsx`):
  - Шапка проекта: иконка/обложка, eyebrow «Проект», антиквенный заголовок, метаданные (`N дочерних · K договоров · M параметров`).
  - Кнопки: «Редактировать» (→ `/sites/:siteId/edit`), «Сравнить очереди» (→ `/compare?parentSiteId=…` — shortcut под частый сценарий «сравни ГП всех очередей этого ЖК», IA §3.2a), «Новый объект» (→ `/sites/new?parentId=…`).
  - 4 KPI: «Готовы», «Требуют внимания» (warning-trend если ≥1), «Среднее удорожание», «Худшее удорожание» (warning-trend если ≥5%). Считаются на клиенте из `sitesApi.listChildren(parentId)`.
  - Таблица дочерних (не карточки, потому что объекты однотипные и хорошо сравниваются построчно): иконка с амбровой обводкой для `attention`, тёплая подсветка строки, чипы `contract_kind`, число параметров, удорожание, `StatusPill`, дата изменения. Drill-down кликом по имени строки на `/sites/:siteId` дочернего — снова промежуточный или карточка по тому же правилу.
- **`SiteCardView`** теперь подкомпонент `SitePage`. Хуки счётчиков (`bundles`, `events`, `comparisons`) поднимаются только для конечных объектов; для промежуточного экрана они не нужны.
- **Хлебные крошки.** Новый общий примитив `Breadcrumbs` (`src/components/ui-domain/Breadcrumbs.tsx`) с `ChevronRight`-разделителями и автоматическим деактивированием последнего сегмента. Цепочка строится в `SitePage.buildBreadcrumbs(site, allSites)` — снизу вверх по `parent_id` через `Map<id, site>`, с защитой от циклов через `Set`. Стартовый сегмент — «Объекты» → `/dashboard`. На дашборде крошек намеренно нет (верхний уровень).
