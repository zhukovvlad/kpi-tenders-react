# Дневник разработки — react-kpi-tenders

## 2026-04-26 — День 5: Синхронизация с артефактной архитектурой Go-бэкенда

### Контекст

Go-бэкенд перешёл на новую модель артефактов: вместо того чтобы хранить `storage_path` готовых файлов прямо в `result_payload`, сервис-воркер теперь регистрирует каждый артефакт как отдельную запись `Document` (с `parent_id` и `artifact_kind`) и кладёт в payload UUID этого документа (`*_document_id`). Скачивание через `GET /api/v1/documents/:id/url?download=true` — единственный механизм.

Фронтенд был синхронизирован с этим контрактом.

---

### Что изменено

**1. `src/types/task.ts` — обновлён `TaskResultPayload`**

Поля `*_storage_path` заменены на `*_document_id`:

```ts
// ДО
export interface TaskResultPayload {
  md_storage_path?: string
  anonymized_storage_path?: string
  entities_map_storage_path?: string
  char_count?: number
  section_count?: number
  entity_count?: number
}

// ПОСЛЕ
export interface TaskResultPayload {
  // convert worker — артефакт зарегистрирован как Document
  md_document_id?: string
  char_count?: number
  section_count?: number
  // anonymize worker — артефакты зарегистрированы как Documents
  anonymized_document_id?: string
  entities_map_document_id?: string
  entity_count?: number
}
```

**2. `src/types/document.ts` — добавлены `ArtifactKind` и поле `artifact_kind`**

```ts
export type ArtifactKind = "convert_md" | "anonymize_doc" | "anonymize_entities"

export interface Document {
  // ...
  parent_id: string | null
  artifact_kind: ArtifactKind | null  // null = загружен пользователем
  // ...
}
```

Правило для рендеринга: если `artifact_kind === null` — документ пользовательский; если не null — артефакт обработки, в `DocumentsPage` не отображается (бэкенд фильтрует `WHERE parent_id IS NULL`).

**3. `src/services/api/tasks.ts` — удалён `getResultUrl`**

Удалены `TaskResultFile` тип и метод `getResultUrl` (вызывал несуществующий эндпоинт `GET /api/v1/tasks/:id/result-url`). Теперь только:

```ts
export const tasksApi = {
  getByDocument: (documentId: string): Promise<Task[]> => ...,
  start: (documentId: string, moduleName: TaskModule): Promise<Task> => ...,
}
```

**4. `src/pages/AnonymizationPage.tsx` — обновлены `DownloadButton` и `ResultDialog`**

`DownloadButton` теперь принимает `documentId` вместо `taskId + file` и использует `documentsApi.getPresignedUrl`:

```tsx
// ДО
function DownloadButton({ taskId, file, ... }) {
  const url = await tasksApi.getResultUrl(taskId, file, true)
}

// ПОСЛЕ
function DownloadButton({ documentId, ... }) {
  const url = await documentsApi.getPresignedUrl(documentId, true)
}
```

`ResultDialog` читает `anonymized_document_id` и `entities_map_document_id` из `result_payload`. Кнопки скачивания показываются только если соответствующий `*_document_id` присутствует в payload.

---

### Архитектурные решения

**Почему `*_document_id` в payload, а не прямой путь?**
- Все артефакты — полноценные `Document`-записи с `parent_id` и `artifact_kind`
- Единый механизм доступа: `GET /api/v1/documents/:id/url` с проверкой прав tenant-изоляции
- Бэкенд никогда не отдаёт `storage_path` напрямую клиенту — только presigned URL через MinIO

**Почему `DocumentsPage` не показывает артефакты?**
- Go `ListRootDocumentsByOrganization` фильтрует `WHERE parent_id IS NULL`
- Артефакты доступны через `GET /api/v1/documents?parent_id=:uuid` — отдельный эндпоинт для будущего использования
