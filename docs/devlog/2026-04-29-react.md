### TD-001: Batch polling — N+1 → 1 запрос

**Проблема.** Каждый `DocumentAnonymizationRow` держал отдельный `useQuery` с polling 3 с. N документов = N параллельных запросов каждые 3 с.

**Решение (бэкенд).** Новый эндпойнт:

```http
GET /api/v1/tasks?document_ids=id1,id2,...
```

SQL `WHERE document_id = ANY($1::uuid[])`, возвращает плоский массив задач для всех переданных документов за один запрос.

**Решение (фронтенд).** Polling вынесен на уровень страницы — один `useQuery` вместо N:

```ts
const { data: allTasks = [], isLoading: isLoadingTasks, isError: isErrorTasks } = useQuery({
  queryKey: ["tasks", "batch", documents.map((d) => d.id).join(",")],
  queryFn: () => tasksApi.getByDocuments(documents.map((d) => d.id)),
  enabled: documents.length > 0,
  staleTime: 0,
  refetchInterval: (query) => {
    const data = query.state.data as Task[] | undefined
    if (!data || data.length === 0) return false
    const hasActive = data.some(
      (t) => t.status === "pending" || t.status === "processing",
    )
    if (hasActive) return 3000
    // Polling продолжается, если convert завершён, но anonymize ещё не создан
    // (бэкенд создаёт его асинхронно после конвертации).
    const docsWithConvertDone = new Set(
      data
        .filter((t) => t.module_name === "convert" && t.status === "completed")
        .map((t) => t.document_id),
    )
    const docsWithAnonymize = new Set(
      data.filter((t) => t.module_name === "anonymize").map((t) => t.document_id),
    )
    if ([...docsWithConvertDone].some((id) => !docsWithAnonymize.has(id))) return 3000
    return false
  },
})
```

Polling останавливается автоматически, когда все задачи в терминальном состоянии (`completed` / `failed`) и для каждого convert-done документа уже есть задача anonymize.

**Группировка O(n) через `useMemo`.**  До рефакторинга каждая строка вызывала `allTasks.filter(t => t.document_id === doc.id)` — O(n×m). Заменено на `Map`:

```ts
const tasksByDocId = useMemo(() => {
  const map = new Map<string, Task[]>()
  for (const task of allTasks) {
    const existing = map.get(task.document_id)
    if (existing) {
      existing.push(task)
    } else {
      map.set(task.document_id, [task])
    }
  }
  return map
}, [allTasks])

// В рендере:
tasks={tasksByDocId.get(doc.id) ?? []}
```

**`isErrorTasks` → строки.** Ошибка загрузки задач теперь прокидывается в каждую строку как `isErrorTasks: boolean`. Кнопка «Запустить» скрыта при ошибке, чтобы не запустить задачу когда неизвестно, была ли она уже создана:

```tsx
{!hasStarted && !isLoadingTasks && !isErrorTasks && (
  <Button onClick={() => startMutation.mutate("convert")}>Запустить</Button>
)}
```

После старта задачи инвалидируется батч-ключ:

```ts
queryClient.invalidateQueries({ queryKey: ["tasks", "batch"] })
```

```ts
// ДО
import apiClient from "./client"

import apiClient from "@/services/api/client"
```

#### ✅ `formatDate`: защита от "Invalid Date"

`new Date(isoString).toLocaleDateString()` возвращал строку `"Invalid Date"` при невалидном ISO. Добавлена проверка:

```ts
export function formatDate(isoString: string): string {
  const date = new Date(isoString)
  if (isNaN(date.getTime())) return "-"
  return date.toLocaleDateString("ru-RU", { ... })
}
```

#### 🚫 CodeRabbit: "удалить `getByDocuments` из `tasksApi`" — false positive

CodeRabbit предложил удалить метод из публичного API модуля. Метод является намеренно экспортированным — использует батч-эндпойнт бэкенда. Отклонено.

### Баг: `Map` из lucide-react затеняет глобальный `Map`

**Симптом.** После добавления `useMemo`-группировки TypeScript выдавал:

```text
'new' expression, whose target lacks a construct signature, implicitly has an 'any' type. ts(7009)
Expected 0 type arguments, but got 2. ts(2558)
```

на строке `new Map<string, Task[]>()`.

**Причина.** В импортах lucide-react присутствовал `Map` (иконка), который затенял глобальный конструктор `Map`:

```ts
import { ..., Map } from "lucide-react"  // ← перекрывает window.Map
```

TypeScript воспринимал `new Map<string, Task[]>()` как попытку вызвать React-компонент с дженерик-аргументами.

**Решение.** Переименовать импорт иконки:

```ts
import { ..., MapIcon } from "lucide-react"
//                ^-- не конфликтует с глобалом
```

**Вывод.** Иконки lucide-react с именами, совпадающими с глобальными JS-идентификаторами (`Map`, `Set`, `Image`, `Screen`, `File`), нужно импортировать с псевдонимом.

### Третий раунд замечаний

#### ✅ `storage_path` удалён из `Document`

Copilot указал, что `Document.storage_path` присутствует в типе как обязательное поле, хотя бэкенд не возвращает его клиенту (devlog 2026-04-26). Поле удалено из интерфейса — все скачивания всегда идут через `documentsApi.getPresignedUrl`. Убрано и из правил CLAUDE.md.

#### ✅ `anonymizeStatus` — `"pending"` в переходном окне

Когда convert завершён, но бэкенд ещё не создал задачу anonymize (асинхронное создание), `anonymizeStatus` был `"idle"` — строка выглядела зависшей без каких-либо управляющих элементов.

Фикс — явный маппинг переходного состояния:

```ts
const anonymizeStatus: StageStatus =
  anonymizeTask?.status ?? (convertStatus === "completed" ? "pending" : "idle")
```

Пользователь видит чип «Анонимизация — pending» вместо серого «idle».

#### ✅ Баннер ошибки задач с кнопкой «Повторить»

`isErrorTasks` использовался только для скрытия кнопки «Запустить», но страница ничего не сообщала пользователю при сбое загрузки задач — документы отображались без управляющих элементов и без объяснений.

Добавлен баннер на уровне страницы:

```tsx
{!isLoading && !isError && documents.length > 0 && isErrorTasks && (
  <div className="...">Не удалось загрузить статусы задач. Управление обработкой недоступно.
    <Button onClick={() => void refetchTasks()}>Повторить</Button>
  </div>
)}
```

`refetch` деструктурирован из `useQuery` как `refetchTasks`.

#### ✅ Оптимистичное обновление кеша при старте задачи

До исправления после `tasksApi.start` запускался только `invalidateQueries` — строка оставалась в состоянии «idle» до завершения фонового рефетча (задержка сети).

Теперь новая задача вставляется в кеш немедленно через `setQueriesData`, а `invalidateQueries` запускает фоновое обновление:

```ts
onSuccess: (data, module) => {
  // Optimistically insert the new task into the batch cache so the row
  // immediately reflects the started state before the refetch completes.
  queryClient.setQueriesData<Task[]>(
    { queryKey: ["tasks", "batch"] },
    (old) => (old ? [...old, data] : [data]),
  )
  queryClient.invalidateQueries({ queryKey: ["tasks", "batch"] })
}
```

## 2026-04-29 — День 7 (продолжение): Extraction Wizard

### Контекст

Реализован полный 4-шаговый Stepper UI для семантического извлечения параметров из тендерных документов. Пользователь задаёт вопросы, выбирает объект и документ, запускает пайплайн и наблюдает за прогрессом в реальном времени.

---

### Новые API эндпойнты (бэкенд)

```http
GET  /api/v1/sites                        → ConstructionSite[]
GET  /api/v1/documents?site_id=:uuid      → Document[] (фильтр по объекту)
POST /api/v1/documents/upload             multipart, поле site_id опциональное
POST /api/v1/documents/:id/extract        body: { questions: string[] } → { task_id: string }
```

Пайплайн модулей: `convert → anonymize → resolve_keys → extract`

Результат задачи `extract`: плоский JSON `{ [ключ]: string | null }`.

### Шаги визарда

**Шаг 1 — `StepQuestions`** — список вопросов с Input + кнопкой добавления. Теги популярных вопросов (срок строительства, стоимость работ, подрядчик, материалы, площадь объекта, требования к качеству, гарантийные обязательства, штрафные санкции) — клик добавляет вопрос. Нельзя перейти дальше без хотя бы одного вопроса.

**Шаг 2 — `StepSite`** — shadcn `Select` с объектами из `sitesApi.list()`. Состояния загрузки и ошибки обработаны.

**Шаг 3 — `StepDocument`** — shadcn `Tabs`:
- «Существующий документ»: таблица документов, отфильтрованных по `selectedSiteId`.
- «Загрузить новый»: react-dropzone зона. Файл загружается сразу после drop, `uploadedDocumentId` сохраняется в контексте.

**Шаг 4 — `StepReviewAndRun`**:
- Саммари: «Вы хотите узнать N параметров для объекта X из документа Y»
- Кнопка «Запустить» → `extractionApi.initiate()` → получает `task_id`
- Polling через `setInterval` вызывает `tasksApi.getByDocuments`
- 4 стадии пайплайна с shadcn `Progress` и chip'ами: Конвертация → Деперсонализация → Анализ структуры → Извлечение данных
- По завершению рендерится `ExtractionResultsTable`

**`ExtractionResultsTable`** — таблица key-value из `ExtractResultPayload` с shadcn `Badge` для статуса значений.

**`ExtractionWizard`** — контейнер stepper'а: индикатор 4 шагов с иконками (активный/выполненный), маршрутизация по шагам, кнопки Назад/Далее с валидацией.
